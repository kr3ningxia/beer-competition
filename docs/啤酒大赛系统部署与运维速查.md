# 啤酒大赛系统部署与运维速查

本文记录当前项目已经跑通的服务器部署方式、CI/CD 流程、数据库同步策略和常用运维命令。新对话或新接手开发时，优先阅读本文了解现状。

## 1. 当前结论

- 主代码仓库：Gitee，远程名为 `origin`。
- CI/CD 仓库：GitHub，远程名为 `github`。
- 服务器：阿里云 ECS，华东 1（杭州），Ubuntu 24.04 64 位，2 vCPU / 4 GiB。
- 当前公网 IP：`121.40.25.17`。
- 当前私网 IP：`172.20.186.110`。
- 当前公网带宽：固定带宽 3 Mbps。
- 当前系统盘：ESSD Entry 云盘 40 GiB。
- 当前实例规格：`ecs.e-c1m2.large`。
- 当前实例到期时间：2027-06-08 23:59:59。
- 部署根目录：`/opt/beer-competition`。
- MySQL：宿主机安装，数据库名 `beer_competition`。
- Nginx：宿主机安装，负责静态资源和 API 反向代理。
- Redis：Docker Compose 管理，容器名 `beer-redis`。
- Spring Boot API：Docker Compose 管理，容器名 `beer-api`。
- 前端：GitHub Actions 构建后上传 dist，由宿主机 Nginx 直接托管。
- 数据库结构变更：暂时手动执行 SQL，不接入自动迁移。

```text

│  ├─ Dockerfile
│  ├─ docker-compose.yml
│  └─ .env

│  └─ release.zip

   ├─ console
```

关键文件：

- `/opt/beer-competition/app/.env`：线上运行配置，包含数据库、Redis、JWT、存储等环境变量。不要提交到代码仓库。
- `/opt/beer-competition/app/docker-compose.yml`：管理 `beer-api` 和 `beer-redis`。
- `/opt/beer-competition/scripts/deploy.sh`：服务器端部署脚本，GitHub Deploy 最终调用它。
- `/opt/beer-competition/nginx/beer-competition.conf`：当前项目 Nginx 配置。

## 5. CI/CD 流程

### 5.1 CI

文件：

```text
.github/workflows/ci.yml
```

触发方式：

- push 到 GitHub `main`。
- pull request 到 `main`。
- GitHub Actions 页面手动运行。

做的事情：

```text
1. 使用 Java 17 构建后端 jar
2. 使用 Node 22 构建 console
3. 使用 Node 22 构建 judge
4. 组装 release.zip
5. 上传 artifact：beer-competition-release
```

CI 只检查构建，不部署服务器。

### 5.2 手动 Deploy

文件：

```text
.github/workflows/deploy.yml
```

触发方式：

```text
GitHub -> Actions -> Deploy -> Run workflow -> main
```

做的事情：

```text
1. GitHub Actions 构建后端和前端
2. 组装 release.zip
3. 使用 SSH 上传到 /opt/beer-competition/uploads/release.zip
4. SSH 执行 /opt/beer-competition/scripts/deploy.sh
5. deploy.sh 备份旧版本、覆盖新版本、重建 API 镜像、重启容器、reload Nginx、做健康检查
```

GitHub Secrets 必须包含：

```text
ECS_HOST
ECS_PORT
ECS_USER
ECS_SSH_KEY
```

`ECS_SSH_KEY` 是完整私钥，必须包含：

```text
-----BEGIN OPENSSH PRIVATE KEY-----
...
-----END OPENSSH PRIVATE KEY-----
```

不要把私钥写入仓库或文档。

### 5.3 服务器配置升级后的参数检查

当前服务器已升级到 2 vCPU / 4 GiB，应用代码中没有写死 CPU 或内存规格，通常不需要因为本次配置升级修改代码配置。需要重点检查以下运行参数：

| 参数或配置 | 是否需要调整 | 说明 |
| --- | --- | --- |
| GitHub Secret `ECS_HOST` | 视公网 IP 是否变化 | 如果部署目标公网 IP 已变为 `121.40.25.17`，必须把 `ECS_HOST` 更新为该 IP；如果原来已经是这个 IP，则不用改。 |
| GitHub Secret `ECS_PORT` | 通常不用改 | 只有 SSH 端口从 22 改成其他端口时才需要调整。 |
| GitHub Secret `ECS_USER` | 通常不用改 | 只有部署用户从 `root` 或原部署用户切换时才需要调整。 |
| GitHub Secret `ECS_SSH_KEY` | 通常不用改 | 只有重装系统、替换部署用户或重建 SSH key 时才需要更新。 |
| DNS 解析 | 视访问方式决定 | 如果后续绑定域名，需要把 A 记录指向 `121.40.25.17`；当前按公网 IP 访问则不用配置 DNS。 |
| 阿里云安全组 | 建议复核 | 确认 80、SSH 端口放行；3306、6379 不应对公网开放。 |
| Nginx 配置 | 通常不用改 | 继续使用公网 IP 访问时不用改；绑定域名后再补 `server_name` 和 HTTPS 配置。 |
| `/opt/beer-competition/app/.env` | 通常不用改 | 数据库、Redis、JWT、短信、存储参数与 ECS 规格无直接关系。 |
| API JVM 内存 | 可选优化 | 4 GiB 内存下可以给 Spring Boot 预留更稳定的堆内存，但不要占满整机内存。 |
| Redis 内存 | 可选优化 | 当前测试阶段数据量小，可保持现状；后续缓存量变大时再设置 Redis 最大内存。 |
| 上传文件存储 | 建议关注容量 | 系统盘 40 GiB，测试阶段够用；如果报名材料、图片、附件增长，优先迁移到 OSS 或加数据盘。 |
| 公网带宽 | 建议关注体验 | 固定 3 Mbps 对后台和评委端页面够用；大量图片、附件下载或并发访问时可能成为瓶颈。 |

如果需要手动给 API 设置 JVM 内存，可以在服务器的 `docker-compose.yml` 中给 `beer-api` 增加环境变量，例如：

```yaml
environment:
  JAVA_TOOL_OPTIONS: "-Xms512m -Xmx1536m"
```

建议先保持当前配置观察资源占用；如果 `docker stats` 或 `free -h` 显示 API 内存波动较大，再设置 `JAVA_TOOL_OPTIONS`。在 4 GiB 机器上，`-Xmx` 建议控制在 `1536m` 到 `2048m` 以内，给 MySQL、Redis、Nginx、Docker 和系统缓存保留空间。

## 6. release.zip 标准结构

无论本地手动打包还是 GitHub Actions 自动打包，都必须满足：

```text
release.zip
├─ app
│  └─ beer-api.jar
└─ www
   ├─ console
   │  ├─ index.html
   │  └─ assets
   └─ judge
      ├─ index.html
      └─ assets
```

服务器检查压缩包：

```bash
unzip -l /opt/beer-competition/uploads/release.zip | head -n 40
```

如果在 Windows 本地压缩，`unzip` 可能提示：

```text
appears to use backslashes as path separators
```

当前 `deploy.sh` 已兼容这种 warning。

## 7. deploy.sh 行为

脚本路径：

```bash
/opt/beer-competition/scripts/deploy.sh
```

手动执行：

```bash
bash /opt/beer-competition/scripts/deploy.sh /opt/beer-competition/uploads/release.zip
```

脚本主要步骤：

```text
1. 校验 release.zip
2. 解压到临时目录
3. 校验 app/beer-api.jar、www/console、www/judge
4. 备份当前 jar 和前端目录到 /opt/beer-competition/backups/release-时间戳
5. 停止 beer-api
6. 覆盖 beer-api.jar
7. 覆盖 console 和 judge dist
8. 修正文件权限
9. docker compose up -d --build beer-api
10. nginx -t
11. systemctl reload nginx
12. 最多等待 60 秒检查 API
13. 检查前端
14. 输出 Deploy success
```

最新日志：

```bash
ls -lt /opt/beer-competition/logs | head
tail -n 120 /opt/beer-competition/logs/deploy-*.log
```

部署成功关键日志：

```text
API health check passed
Check frontend
Deploy success
Backup saved at: /opt/beer-competition/backups/release-时间戳
```

## 9. 数据库同步策略

当前数据库不自动迁移。涉及表结构或线上数据变更时，按手动 SQL 流程执行。

### 9.1 表结构变更

推荐在仓库中新增 SQL 文件：

```text
docs/sql/20260609_add_xxx_field.sql
```

上传到服务器：

```text
/opt/beer-competition/uploads/20260609_add_xxx_field.sql
```

执行前备份：

```bash
mysqldump -uroot -p --default-character-set=utf8mb4 --single-transaction beer_competition > /opt/beer-competition/backups/beer_competition_$(date +%Y%m%d%H%M%S).sql
```

执行 SQL：

```bash
mysql -uroot -p --default-character-set=utf8mb4 beer_competition < /opt/beer-competition/uploads/20260609_add_xxx_field.sql
```

验证结构：

```bash
mysql -uroot -p beer_competition -e "SHOW COLUMNS FROM 表名;"
```

### 9.2 局部同步指定表

适合字典表、风格库、系统配置表。先确认线上已有数据是否可以覆盖。

本地导出指定表：

```powershell
mysqldump -hlocalhost -P3306 -uroot -p本地密码 --default-character-set=utf8mb4 --single-transaction beer_competition 表名1 表名2 --result-file=C:\Users\a1508\Desktop\partial.sql
```

服务器导入前仍需备份线上库。

### 9.3 全量同步

全量同步会覆盖线上数据，只适合确认线上数据可丢弃或测试环境。

本地导出：

```powershell
mysqldump -hlocalhost -P3306 -uroot -p本地密码 --default-character-set=utf8mb4 --single-transaction --routines --triggers beer_competition --result-file=C:\Users\a1508\Desktop\beer_competition_full.sql
```

服务器导入前必须先备份：

```bash
mysqldump -uroot -p --default-character-set=utf8mb4 --single-transaction --routines --triggers beer_competition > /opt/beer-competition/backups/beer_competition_before_full_import_$(date +%Y%m%d%H%M%S).sql
```

导入：

```bash
mysql -uroot -p --default-character-set=utf8mb4 beer_competition < /opt/beer-competition/uploads/beer_competition_full.sql
```

## 11. 常见问题

### 11.1 GitHub Actions 读不到 Secrets

现象：

```text
Bad port ''
ssh-keyscan -p "" ""
```

原因：

```text
ECS_HOST、ECS_PORT、ECS_USER、ECS_SSH_KEY 没有按独立 Repository secrets 创建
```

处理：

```text
GitHub -> Settings -> Secrets and variables -> Actions -> Repository secrets
```

确认存在：

```text
ECS_HOST
ECS_PORT
ECS_USER
ECS_SSH_KEY
```

### 11.2 SSH 私钥格式错误

现象：

```text
Load key "/home/runner/.ssh/ecs_deploy_key": error in libcrypto
Permission denied (publickey)
```

原因：

```text
复制了公钥 .pub
私钥缺少 BEGIN 或 END
多行私钥被粘成一行
服务器 authorized_keys 没有对应公钥
```

处理：

```bash
ssh-keygen -t ed25519 -C "github-actions-beer-deploy" -f /root/.ssh/github_actions_beer -N ""
cat /root/.ssh/github_actions_beer.pub >> /root/.ssh/authorized_keys
chmod 700 /root/.ssh
chmod 600 /root/.ssh/authorized_keys
cat /root/.ssh/github_actions_beer
```

把完整私钥更新到 GitHub `ECS_SSH_KEY`。部署跑通后删除服务器私钥：

```bash
rm -f /root/.ssh/github_actions_beer
```

### 11.3 API 502 或接口不通

检查容器：

```bash
docker compose -f /opt/beer-competition/app/docker-compose.yml ps
cd /opt/beer-competition/app
docker compose logs --tail=200 beer-api
```

检查本机 API：

```bash
curl -i http://127.0.0.1:8080/api/portal/public/home
```

检查 Nginx：

```bash
nginx -t
systemctl status nginx --no-pager
```

### 11.4 前端空白页或静态资源 404

检查前端构建 base：

```bash
npm run build -- --base=/console/
npm run build -- --base=/judge/
```

检查服务器文件：

```bash
ls -lh /opt/beer-competition/www/console
ls -lh /opt/beer-competition/www/judge
grep -o "assets/[^\"']*" /opt/beer-competition/www/console/index.html
grep -o "assets/[^\"']*" /opt/beer-competition/www/judge/index.html
```

检查 Nginx 静态资源：

```bash
curl -I http://127.0.0.1/console/
curl -I http://127.0.0.1/judge/
```

### 11.5 GitHub 推送失败

现象：

```text
Failed to connect to github.com port 443
Recv failure: Connection was reset
```

处理：

```powershell
Test-NetConnection github.com -Port 443
```

如果 443 不通，检查代理或网络。配置 Git 代理示例：

```bash
git config --global http.proxy http://127.0.0.1:代理端口
git config --global https.proxy http://127.0.0.1:代理端口
```

## 12. 发布检查清单

```text

4. push origin main 到 Gitee
5. push github main 到 GitHub

```
