## 1. 结论摘要

本系统定位为啤酒大赛全流程数字化管理工具，覆盖厂商报名、比赛评审、数据汇总、结果查询四个环节。业务上由三端组成：厂商 Portal、评审 H5、主办方后台；工程实现以当前项目为准：Spring Boot 后端、Vue 桌面端、Vue H5 评审端三个项目，共用 MySQL 数据库。

当前仓库已经具备基础骨架：后台登录、比赛创建、评审列表、评审分配、评分配置、厂商酒款提交与列表、二维码标签下载、评审登录、扫码或短编号查询、评分提交、桌长汇总和晋级标记。距离完整目标还需要补齐微信支付、评审注册、状态流转、结果发布、反馈查询、奖项管理、数据导出、评审桌配置页面和更细的权限/校验。会议额外明确了多轮评审、后续轮选择排序、评审桌酒款分配、现场公开看板、二维码短编号兜底、专业评分表备注框等需求，需要进入近期设计。

本次需求应优先围绕 8 月比赛可用性推进。根据 Word PRD 的 Phase 1 和会议讨论，MVP 重点放在评审 H5 和主办方后台：评审注册/登录、扫码或短编号查看酒款、评分提交、桌长共识分、每桌晋级数量校验、评审与酒款编排、进度看板、评分维度配置、后续轮选择排序。厂商报名与支付在 Word 中被列入 Phase 2；会议中也倾向近期先完成评审核心闭环，厂商报名与支付除非本次比赛必须线上承接，否则可排在后续阶段。

## 4. 总体业务流程

### 4.1 厂商参赛流程

| 步骤 | 业务动作 | 系统要求 |
| --- | --- | --- |
| 1 | 厂商注册/登录 | 手机号注册、短信验证码登录，记录姓名、手机号、微信号和厂牌信息 |
| 2 | 选择比赛 | 同一厂商账号可参加多场比赛 |
| 3 | 提交酒款 | 同一厂商账号可提交多款参赛酒 |
| 4 | 填写报名表 | 填酒名、投递组别、基础风格、ABV、简介、额外字段 |
| 5 | 生成匿名编码和短编号 | 提交后系统生成唯一匿名编码，并生成扫码失败时可输入的短编号 |
| 6 | 微信支付 | 按比赛配置的报名费付款 |
| 7 | 下载二维码 | 支付成功后可下载二维码，用于现场张贴 |
| 8 | 状态追踪 | 厂商查看酒款当前状态 |
| 9 | 查看反馈 | 结果发布后查看评分、备注、桌长综合评语、奖项 |
| 10 | 获奖后处理 | 下载奖状，填写奖状奖牌收件地址 |

### 4.2 评审现场流程

| 步骤 | 操作 | 说明 |
| --- | --- | --- |
| 1 | 登录 | 手机号 + 验证码 |
| 2 | 选择比赛 | 从本账号被分配的比赛列表选择；只有一场时可自动进入 |
| 3 | 查看角色 | 系统显示当前比赛角色和所在评审桌 |
| 4 | 扫码 | 扫描酒款二维码，获取匿名编码、投递组别、基础风格、简介和允许公开的额外字段；扫码失败时输入短编号 |
| 5 | 评分 | 根据角色对应评分表填写各维度分数 |
| 6 | 写备注 | 跨界评审备注合计不少于 50 字，专业评审备注合计不少于 30 字 |
| 7 | 提交评分 | 提交后允许在桌长汇总前修改 |
| 8 | 桌长汇总 | 桌长查看本桌评分，讨论后填写共识分和综合评语 |
| 9 | 标记晋级 | 每轮每桌完成后，桌长按后台配置勾选固定数量的晋级酒 |
| 10 | 后续轮选择排序 | 第二轮及之后从晋级酒款中选择并排序，形成组别金银铜和总冠军候选 |

### 4.3 主办方管理流程

| 阶段 | 主办方动作 | 系统要求 |
| --- | --- | --- |
| 赛前配置 | 创建比赛、配置报名费、报名截止时间、组别、评分维度 | 后台可配置，评分维度按比赛独立维护 |
| 报名管理 | 查看酒款、状态变更、支付/退款处理 | 支持手动更新状态，支付成功自动更新 |
| 评审准备 | 创建评审桌、维护评审账号、分配角色和桌号 | 每场比赛独立配置，桌长有且只有 1 人 |
| 现场监控 | 查看评分进度、晋级情况、每位评审评分数量、平均耗时和备注字数 | “已评审”按桌长已汇总定义，现场看板需支持公开投屏 |
| 结果输出 | 设置获奖酒款、发布结果、导出 Excel | 发布后厂商可查看反馈和奖项 |

### 4.4 每场比赛独立配置项

比赛配置采用“默认模板 + 单场覆盖”的口径。系统可内置报名表、评分表和奖项规则模板；主办方创建比赛时复制默认模板，并可针对本场比赛调整字段、选项、分值和规则。

| 配置项 | 是否每场独立配置 | 说明 |
| --- | --- | --- |
| 报名费 | 是 | 每场比赛可设置不同金额 |
| 报名表模板 | 是 | 默认模板带入后可调整字段、必填、排序和展示范围 |
| 投递组别 | 是 | 报名表下拉选项，每场比赛的组别列表不同 |
| 基础风格 | 是 | 全局标准风格库，约 200 个风格，报名时必填、单选、可搜索 |
| 报名表额外字段 | 是 | 支持文本、长文本、数字、下拉等字段类型，字段选项可配置 |
| 评分表模板 | 是 | 专业评审使用固定 5 维度模板；跨界评审按比赛配置 2-3 个维度 |
| 角色评分维度 | 是 | 跨界评审、专业评审、桌长分别配置评分字段和备注框 |
| 评分维度分值 | 是 | 每项满分可配置，系统校验总分口径，默认总分 50 |
| 评审桌和桌号 | 是 | 每场比赛创建自己的 A 桌、B 桌等 |
| 每桌人数和角色人数 | 是 | 每张桌可不同，桌长必须有且只有 1 人 |
| 轮次/评审桌酒款 | 是 | 按轮次和评审桌维护酒款、人员和晋级数量，支持后续轮 2A、2B、3A 等桌号 |
| 奖项规则 | 建议支持单场覆盖 | 默认每组别金银铜 + 全场总冠军，单场可调整奖项名称、数量和启用状态 |

### 5.3 匿名编码、短编号与二维码

| 需求 | 说明 | 当前情况 |
| --- | --- | --- |
| 自动生成唯一匿名编码 | 酒款提交后生成匿名编码 | 数据库已有 `beer_entry.uuid`，当前种子数据为 `BC-2026-IPA-0001` |
| 短编号生成 | 为二维码下方生成可人工输入的短编号 | 已由 `entry_scan_label.short_code` 承载 |
| 二维码生成 | 扫码令牌用于生成二维码，标签下方同步展示短编号 | 已由 Portal 标签页生成真实二维码 |
| 厂商下载二维码 | 支付完成后可下载，用于打印和确认 | 已支持单个标签下载和打印 |
| 主办方获取二维码 | 主办方可查看、下载、批量导出二维码，用于比赛当天贴到酒瓶上 | 后台酒款管理页面未实现 |
| 主办方可查看匿名编码和短编号 | 便于现场管理和核对酒款 | 后台酒款列表已返回标签编码和短编号，专门的二维码管理页仍需补齐 |
| 厂商可查看匿名编码 | 便于打印和确认 | Portal 列表和标签页已显示标签编码 |
| 评审短编号输入 | 扫码失败时手动输入短编号进入酒款页 | 已支持 |

现场使用口径：每个参赛酒款生成一个二维码，比赛当天由主办方或现场执行人员将二维码贴到对应酒瓶上。评审通过 H5 扫描瓶身二维码获取匿名酒款信息；如扫码失败，可手动输入二维码下方短编号。

编号匿名性补充：当前采用 `entry_scan_label` 管理评审现场标签，独立保存标签编码、短编号和扫码令牌；`beer_entry.uuid` 继续作为系统内匿名作品编号。会议讨论过“厂商端编号”和“评审端编号”做双编号映射，以降低厂商通过已知编号影响评审的风险，后续如业务确认需要，可基于标签表继续扩展换码、作废和重打能力。

### 6.4 评分表配置

评分表采用默认模板。系统内置跨界评审、专业评审、桌长三类评分表模板；创建比赛时复制模板，主办方可在当前比赛下调整维度、说明、满分和排序。页面和后端校验均按比赛配置动态加载，不能在前端写死维度。

Word 中评分表信息：

| 评分表类型 | 维度 | 各项满分 | 总分口径 |
| --- | --- | --- | --- |
| 大众/跨界评审 | 每场比赛自定义，约 2-3 项 | 每个比赛不一样 | 满分 50 |
| 专业评审 | 香气、外观、味道、口感、整体印象 | 香气 12、外观 3、味道 20、口感 5、整体印象 10 | 满分 50 |
| 桌长总结 | 共识分 + 综合文字反馈 | 桌长查看原始评分后填写独立共识分 | 满分 50 |

批注补充：

| 批注点 | 需求结论 |
| --- | --- |
| 专业评分维度 | 会议确认第一版专业评审使用固定 5 维度和固定分值 |
| 跨界评分维度 | 每个比赛不一样，评分配置必须按比赛独立维护 |
| 满分 50 | 不论 2 个或 3 个维度，总分口径统一为 50 |
| 桌长汇总评分 | 桌长填写共识分，独立于桌长本人评分、组员评分和平均分 |

当前系统已有 `competition_score_config.dimensions_json`，支持按比赛和角色维护评分维度；后台评分配置页面目前是 JSON 文本编辑，后续建议改为表单化配置。

评分表字段配置要求：

| 配置能力 | 说明 |
| --- | --- |
| 适用角色 | 跨界评审、专业评审、桌长分别维护 |
| 维度名称 | 如香气、外观、味道、口感、整体印象等 |
| 维度说明 | 给评审看的评分提示，可选 |

### 6.6 桌长汇总与晋级

| 需求 | 说明 | 当前情况 |
| --- | --- | --- |

桌长汇总暂定流程：

| 步骤 | 说明 |
| --- | --- |
| 1 | 同桌评审先品酒并提交自己的评分表和文字反馈 |
| 2 | 桌长查看每位评审填写的完整评分表、总分和评语 |
| 3 | 本轮本桌酒款全部评完后，桌长进入批次汇总视图 |
| 4 | 桌长按酒款查看原始评分，填写共识分 |
| 5 | 桌长填写综合评语 |
| 6 | 桌长按本桌配置数量勾选晋级酒款，例如选且只能选 3 款 |
| 7 | 桌长提交汇总结果；结果锁定前允许继续调整 |

待确认点：桌长综合评语是否需要至少 30 字；桌长提交后由哪个动作锁定；主办方是否可在后台修改桌长结果。当前数据模型只有酒款级 `is_advanced`，建议补充轮次/评审桌/酒款关系，并在后台和桌长端按评审桌配置做晋级数量校验。

## 11. Word 批注逐条处理结论

| 序号 | 批注锚点 | 批注内容 | 需求处理 |
| --- | --- | --- | --- |
| 1 | 支付成功后，酒款状态自动更新为「已报名」 | 建议类似：待付款、已付款/报名成功、已取消/已经退款 | 状态流转采用 `PENDING_PAYMENT`、`REGISTERED`、`CANCELED`，并保留 `STORED`、`RESULT_PUBLISHED` |
| 2 | 数量（每组） | 每个比赛的两种评审角色数量可能不同，有时每桌三人，有时每桌四人 | 评审桌需要支持角色人数配置，不能写死 1+2+1 |
| 3 | 3.4 评分表维度 | 当前维度是草稿，后续会补充详细版 | 评分维度必须后台配置，当前示例不作为最终规则 |
| 4 | 提交桌长共识分 | 要加“是否晋级”，每轮每桌按后台配置选且只能选 N 款晋级酒 | 当前已有单酒款晋级标记，后续补轮次和数量校验 |
| 5 | 创建评审桌 | 用“评审桌”、“桌号”，避免和酒款投报“组别”混淆 | 全文和系统文案统一为评审桌、桌长、桌号 |
| 6 | 每桌组成 | 不同比赛或同比赛不同小组人数不同，唯一确定是桌长有且只有 1 人 | 后台增加桌级规则或分配校验，保证每桌一个桌长 |
| 7 | 评分进度 | “已评审”的定义：桌长已汇总 | 看板统计以 `is_final = true` 为已评审口径 |
| 8 | 排名展示 + 选出前 N 名 | 非必须，不同桌打分手感不同，分数排名只对同一桌有意义 | 排名降级为辅助功能，优先做进度、晋级、结果确认 |
| 9 | 大众评审评分维度待确认 | 每个比赛不一样，但满分 50，分为 2 或 3 个维度 | `ScoreConfig` 按比赛配置，总分校验 50 |
| 10 | 专业评审评分维度待确认 | 已补充 | 以业务后续详细评分表为准 |
| 11 | 投报组别待确认 | 每个比赛不一样，需为每个比赛自定义 | `competition_category` 按比赛维护，后台补配置入口 |
| 12 | 桌长综合调整规则待确认 | 桌长查看每位评审填写的信息后，填写共识分、综合评语并勾选晋级 | 共识分独立于个人评分和平均分；何时锁定待确认 |
| 13 | 获奖类型待确认 | 每个投递组别前三名为金银铜；每场比赛有一个全场总冠军，从各组别的金奖作品中评出 | 新增默认奖项模板、单场覆盖和结果发布能力 |
| 14 | 二维码获取方式 | 主办方会把二维码贴到酒瓶上，评审比赛当天扫描瓶身二维码 | 后台需要支持主办方查看、下载、批量导出二维码 |
| 15 | 厂商可见反馈 | 厂商可查看每位评审原始评分表全部内容、桌长总结和最终得奖情况 | Portal 结果页需展示完整评分反馈，且只能查看本厂牌酒款 |
