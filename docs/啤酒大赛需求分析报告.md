## 1. 结论摘要

系统当前已经从早期骨架推进到可串联的测试版本。三端分别为厂商 Portal、评审 H5、主办方后台；工程实现以当前仓库为准：Spring Boot 后端、Vue 桌面端、Vue H5 评审端，共用 MySQL 数据库。

当前已具备的主链路：

| 模块 | 当前已具备能力 |
| --- | --- |
| 厂商 Portal | 公开赛事、赛事详情、公开结果、短信登录、我的参赛、提交酒款、模拟支付、下载现场参赛标签、提交送样信息、查看个人结果、下载证书 |
| 评审 H5 | 短信登录、评审注册、资料完善与审核状态、任务列表、扫码或短编号解析、动态评分表、整数评分、备注字数校验、我的评分记录、桌长汇总、同桌确认、后续轮排序 |
| 主办方后台 | 登录、比赛创建与详情工作台、组别/风格/报名字段/评分表/评审桌配置、评审账号审核与编排、酒款管理、状态处理、轮次编排、进度查看、奖项生成与确认、证书上传、结果发布、评分数据导出 |

当前仍需优先完善的部分：

| 模块 | 剩余缺口 |
| --- | --- |
| 支付 | 目前为模拟支付和后台手动确认，真实微信支付、回调、退款、对账、发票未接入 |
| 厂商结果后处理 | 奖牌/奖状收件地址未实现 |
| 二维码运营 | 厂商单个参赛标签下载已支持；后台批量参赛标签支持 A4 四联 PDF 和图片文件包 |
| 现场看板 | 已有后台进度和现场看板入口，平均耗时、备注字数质量、大屏口径需要验收和补强 |
| 奖项规则维护 | 已有默认规则和奖项结果流程，奖项规则的可视化维护和并列奖项口径仍需确认 |
| 权限和锁定 | 轮次锁定、结果发布、后台覆盖确认已有接口，具体业务授权边界需定稿 |
| 测试验收 | 三端需要按真实比赛数据做全流程验收，尤其厂家端报名/付款/送样、第一轮评分、后续轮排序、奖项发布 |

| 业务术语 | 当前枚举 | 说明 |
| --- | --- | --- |

## 4. 总体业务流程

### 4.1 厂商参赛流程

| 步骤 | 业务动作 | 当前系统口径 |
| --- | --- | --- |
| 1 | 浏览赛事 | 未登录可看公开首页、赛事列表、赛事详情、公开结果 |
| 2 | 登录 | 厂商联系人使用手机号和验证码登录 |
| 3 | 完善资料 | 维护联系人姓名、手机号、微信号、厂牌信息 |
| 4 | 选择比赛 | 只允许对开放报名的比赛提交酒款 |
| 5 | 提交酒款 | 填写酒名、投递组别、基础风格、ABV、额外字段 |
| 6 | 生成编号 | 提交后生成系统匿名编号、标签编码、短编号和扫码令牌 |
| 7 | 支付报名费 | 当前支持模拟支付和后台手动确认；真实微信支付待接入 |
| 8 | 下载标签 | 付款确认后下载现场参赛标签，标签含二维码、参赛编号和组别 |
| 9 | 提交送样信息 | 按赛事送样方式填写快递或现场送样信息 |
| 10 | 追踪状态 | 查看付款、送样、入库、结果发布等节点 |
| 11 | 查看结果 | 结果发布后查看评分、备注、桌长意见、轮次结果、奖项和证书 |

### 4.2 比赛状态流

| 状态 | 含义 | 典型动作 |
| --- | --- | --- |
| `DRAFT` | 草稿 | 创建比赛、配置基础信息、组别、报名字段、评分表、评审桌 |
| `REGISTRATION_OPEN` | 报名开放 | 厂商可提交酒款 |
| `REGISTRATION_CLOSED` | 报名关闭 | 停止新报名，主办方核对酒款与收样 |
| `JUDGING_PREP` | 评审准备 | 配置评审、酒款分桌和第一轮 |
| `JUDGING` | 评审中 | 发布轮次，评审扫码评分、桌长汇总和排序 |
| `RESULT_CONFIRMING` | 结果确认 | 生成奖项草稿、确认奖项、上传证书 |
| `PUBLISHED` | 已发布 | 厂商可查看结果和证书 |
| `ARCHIVED` | 已归档 | 赛后归档，保留历史数据 |

### 4.3 主办方管理流程

| 阶段 | 主办方动作 | 当前系统口径 |
| --- | --- | --- |
| 赛前配置 | 创建比赛、设置报名窗口、费用、早鸟价、简介、送样、组别、风格库、报名字段、评分表、评审桌 | 已有比赛创建和详情工作台 |
| 报名管理 | 查看酒款、筛选状态、编辑报名信息、确认付款、确认入库、取消报名 | 已有酒款管理、风险提示、修改原因和操作日志 |
| 评审准备 | 维护评审账号、审核资料、配置基础评审桌、分配角色 | 已有评审列表、状态审核、手机号更正和比赛编排 |
| 轮次编排 | 创建第一轮、分配评审桌成员和酒款、发布轮次、创建后续轮 | 已有第一轮和后续轮接口/页面 |
| 现场评审 | 监控评分进度、确认桌长结果、必要时后台覆盖确认 | 已有进度和同桌确认覆盖 |
| 结果发布 | 生成奖项草稿、确认奖项、上传证书、发布结果、导出评分数据 | 已有奖项和导出主链路 |

### 5.2 公开赛事与报名

| 能力 | 当前情况 |
| --- | --- |
| 公开首页 | 展示开放赛事、赛事阶段、报名入口和结果入口 |
| 赛事列表 | 展示报名窗口、比赛时间、费用、早鸟价等 |
| 赛事详情 | 展示赛事简介、组别、基础风格库、报名字段、送样和结果说明 |
| 报名入口 | 仅 `REGISTRATION_OPEN` 状态可提交酒款 |
| 报名字段 | 酒名、投递组别、基础风格、ABV、额外字段 |
| 字段来源 | 投递组别来自比赛组别，基础风格来自比赛绑定风格库，额外字段来自 `entry_field_config` |

酒款简介不作为固定字段。需要补充特殊原料、工艺、桶陈等信息时，通过额外字段配置承接，并由“评审可见”配置决定是否在评审端展示。

### 5.4 酒款状态

酒款主状态、付款状态、送样状态是三条相关但不同的状态线。

| 状态线 | 编码 | 含义 |
| --- | --- | --- |
| 酒款状态 | `PENDING_PAYMENT` | 已提交资料，待付款 |
| 酒款状态 | `REGISTERED` | 付款确认，报名成功 |
| 酒款状态 | `STORED` | 主办方确认酒样入库 |
| 酒款状态 | `CANCELED` | 报名取消 |
| 酒款状态 | `RESULT_PUBLISHED` | 结果已发布到厂商端 |
| 付款状态 | `UNPAID`、`PAID`、`CANCELED`、`REFUNDED` | 付款与退款处理 |
| 送样状态 | `NOT_SUBMITTED`、`SUBMITTED`、`RECEIVED` | 厂商提交送样信息和主办方收样 |

### 5.5 匿名编码与二维码

| 编号 | 承载位置 | 用途 |
| --- | --- | --- |
| 系统匿名编号 | `beer_entry.uuid` | 系统内酒款匿名标识 |
| 标签编码 | `entry_scan_label.label_code` | 后台核对和内部追踪 |
| 短编号 | `entry_scan_label.short_code` | 现场参赛标签展示为参赛编号，扫码失败时可手动输入 |
| 扫码令牌 | `entry_scan_label.scan_token` | 二维码跳转和评审端解析 |

现场参赛标签展示二维码、参赛编号（短编号）和组别。评审端扫码后展示匿名编码、组别、风格、ABV、评审可见额外字段、比赛、轮次和评审桌，不展示酒名、厂商和联系人信息。

### 6.3 评分表和校验

| 要求 | 当前情况 |
| --- | --- |
| 按比赛和角色加载评分表 | 已支持 `competition_score_config` |
| 专业评审 5 维度 | 当前默认口径：香气 12、外观 3、味道 20、口感 5、整体印象 10 |
| 跨界评审 2-3 维度 | 按比赛配置 |
| 总分 50 | 配置和提交均按 50 分口径 |
| 整数评分 | 后端校验整数，前端使用整数步进 |
| 每维度备注 | 前端按配置生成备注框 |
| 备注字数 | 后端校验 `commentMinLength`，专业/跨界可分别配置 |
| 桌长综合评语 | 后端按桌长评分配置校验最小字数 |
| 修改边界 | 桌长已汇总或本桌结果已确认后，个人评分锁定 |

### 6.4 第一轮桌长汇总

| 步骤 | 当前系统口径 |
| --- | --- |
| 1 | 桌长也先完成自己的专业评分 |
| 2 | 同桌评审提交个人评分 |
| 3 | 桌长查看同桌完整评分、维度分、总分和备注 |
| 4 | 桌长填写共识分和综合评语 |
| 5 | 桌长按本桌目标数量勾选晋级 |
| 6 | 同桌评审进入确认页核对共识分、评语和晋级名单 |
| 7 | 确认完成后本桌结果进入主办方确认 |
| 8 | 主办方可在后台覆盖确认或锁定轮次 |

同桌确认是当前系统已经实现的正式流程，应纳入现场操作说明。

### 6.5 后续轮排序

后续轮使用 `RANKING` 类型，通常不再逐项打分。当前支持的目标模式：

| 目标模式 | 用途 |
| --- | --- |
| `TOP_N` | 从候选中选出前 N 名 |
| `MEDALS` | 排出金奖、银奖、铜奖槽位 |
| `CHAMPION` | 从候选中选出总冠军 |

桌长可提交正式排序，参与评审可保存参考排序草稿。主办方锁定后续轮后，可据此生成奖项草稿。

## 11. Word 批注处理结论

| 序号 | 批注锚点 | 当前处理 |
| --- | --- | --- |
| 1 | 酒款状态 | 采用 `PENDING_PAYMENT`、`REGISTERED`、`STORED`、`CANCELED`、`RESULT_PUBLISHED`，付款状态单独维护 |
| 2 | 每桌人数 | 系统按评审桌成员编排，不写死人数；桌长唯一性需要持续校验 |
| 3 | 评分维度 | 按比赛和角色配置，专业评审默认 5 维度，跨界评审可配置 |
| 4 | 是否晋级 | 第一轮按轮次评审桌目标数量勾选，后续轮按排序目标确认 |
| 5 | 评审桌术语 | 文档统一使用评审桌、桌号、桌长 |
| 6 | 评分进度 | 个人评分和桌长结果分开统计，已评审以桌长结果/轮次结果为关键口径 |
| 7 | 排名 | 跨桌分数只作参考，正式奖项通过后续轮排序和主办方确认 |
| 8 | 二维码 | 厂商单个参赛标签、后台批量参赛标签 PDF 和图片文件包已支持 |
| 9 | 厂商反馈 | 已有个人结果页，发布后展示评分、备注、桌长意见、轮次结果、奖项和证书 |
