### 一、啤酒大赛完整流程

本文档按当前需求分析报告和会议讨论整理比赛从赛前配置到赛后发布的业务流程。会议已明确的口径直接进入流程；仍未定稿的部分集中放在“待确认事项”。

#### 阶段 1：赛前筹备与比赛配置

1. **创建比赛基础信息**

   主办方在后台创建比赛，设置比赛名称、比赛日期、报名截止时间、报名费金额、比赛状态等基础信息。

2. **从默认模板生成本场配置**

   系统内置报名表模板、评分表模板、奖项规则模板。创建比赛时复制默认模板到当前比赛，后续主办方可按本场比赛调整，不影响其他比赛和历史比赛。

3. **配置报名表**

   主办方配置本场比赛报名表字段，包括酒名、投递组别、基础风格、ABV、酒款简介和额外字段。

   - 投递组别是报名表下拉选项，每场比赛单独维护，可理解为本场比赛的赛道，例如创意拉格、浅色拉格、深色拉格。
   - 基础风格是酒款信息的一部分，必填、单选。系统应内置约 200 个标准啤酒风格，厂商通过搜索或下拉选择。
   - 基础风格默认使用全局标准风格库；如风格指南更新，由后台统一维护风格库。
   - 额外字段可配置字段类型、下拉选项、是否必填、展示顺序、是否在评审扫码页展示。

4. **配置评分表**

   主办方按角色配置评分表：跨界评审、专业评审、桌长分别维护。

   - 专业评审第一版固定 5 个维度：香气 12 分、外观 3 分、味道 20 分、口感 5 分、整体印象 10 分，总分 50 分。
   - 跨界评审每场比赛可配置 2-3 个维度，维度名称、说明、满分和排序均可调整，总分 50 分。
   - 每个评分维度下方都需要一个备注框。专业评审为香气备注、外观备注、味道备注、口感备注、整体印象备注；跨界评审按配置维度生成备注框。
   - 单个备注框可选填，但专业评审所有备注合计不少于 30 字，跨界评审所有备注合计不少于 50 字。
   - 评分必须为整数，不允许小数；前端评分控件按整数步进。
   - 维度说明作为灰色提示文案展示；维度满分放在名称后，例如“香气（12分）”。

5. **配置评审账号与评审桌**

   主办方维护评审账号信息，包括姓名、手机号、微信号、资质信息等。每场比赛单独创建评审桌，例如 A 桌、B 桌。

   - 评审角色与单场比赛绑定，同一评审在不同比赛可担任不同角色。
   - 每张评审桌可配置不同人数。
   - 每张评审桌必须有且只有 1 名桌长。
   - 每桌可配置 N 名跨界评审和 N 名专业评审，具体人数按比赛实际情况设置。

6. **配置轮次、评审桌与酒款**

   主办方按轮次配置评审桌、评审人员、酒款范围和晋级数量。第一轮通常使用 A 桌、B 桌等桌号；后续轮可使用 2A、2B、3A 等桌号，便于现场区分。

   - 后台在每张桌中同时配置评审人员和酒款。
   - 配置酒款时，先选择投递组别，系统列出该组别下已报名酒款。
   - 支持手动勾选、拖动、全选，也支持按数量随机分配到某张桌。
   - 如果某个组别酒款数量较多，可手动拆分到多张桌。
   - 酒款池需要支持按组别、是否晋级筛选，并支持按编号搜索。
   - 每张桌可单独配置晋级数量，例如 A 桌必须选 3 款，B 桌必须选 5 款；桌长端需要校验“选且只能选 N 款”。

7. **配置奖项规则**

   系统默认奖项规则为：每个投递组别设置金奖、银奖、铜奖；每场比赛设置 1 个全场总冠军。主办方可按比赛覆盖默认规则，调整奖项名称、数量、排序和启用状态。

#### 阶段 2：厂商报名参赛

1. **厂商注册/登录**

   厂商联系人使用手机号注册并短信验证码登录，填写联系人姓名、手机号、微信号和厂牌信息。

2. **选择比赛并提交酒款**

   厂商选择目标比赛，按本场比赛报名表填写酒款信息。同一厂商账号可参加多场比赛，也可在同一比赛提交多款参赛酒。

3. **生成匿名编码和短编号**

   酒款提交后，系统为该酒款生成唯一匿名编码。评审现场通过二维码扫描匿名编码，不直接看到酒名和厂商信息。

   二维码下方必须同步显示一串可手动输入的短编号，作为扫码失败时的兜底。编号应尽量随机，避免从编号中识别厂商或酒款。

4. **支付报名费**

   厂商按本场比赛配置的报名费付款。支付成功后，酒款状态更新为“已付款，报名成功”。支付失败或超时可重新发起。

5. **查看状态和二维码**

   厂商可在 Portal 查看自己提交酒款的状态和匿名编码。支付成功后，厂商可下载二维码用于确认和打印。

#### 阶段 3：酒款入库与现场准备

1. **主办方接收酒款**

   主办方线下接收参赛酒款，并在后台核对报名信息、匿名编码、短编号和酒款状态。

2. **更新入库状态**

   酒款收到后，主办方在后台将状态更新为“已入库”。如报名取消或退款，状态更新为“已取消”。

3. **准备二维码**

   主办方可在后台查看、下载、批量导出二维码。比赛当天，主办方或现场执行人员将二维码贴到对应酒瓶上。

   二维码标签需要同时包含二维码和短编号。现场倒酒或扫码时，如果瓶身弧面、光线或标签材质导致扫码失败，评审可手动输入短编号进入同一酒款页面。

4. **核对评审桌和轮次**

   现场开始前，主办方核对评审桌、评审人员、酒款批次和二维码，确保评审扫码后进入正确的匿名酒款信息页。

#### 阶段 4：现场评审

1. **评审登录**

   评审使用手机号 + 验证码登录。登录后从自己被分配的比赛列表中选择本场比赛；如果只有一场分配比赛，可自动进入。

2. **查看角色和评审桌**

   系统展示评审在当前比赛中的角色、评审桌和相关提示。角色包括跨界评审、专业评审、桌长。

3. **扫码获取匿名酒款信息**

   评审扫描瓶身二维码，系统根据匿名编码展示匿名酒款信息。

   评审可见信息：

   | 可展示 | 说明 |
   | --- | --- |
   | 匿名编码 | 二维码对应编码 |
   | 投递组别 | 当前比赛配置的组别 |
   | 基础风格 | 当前报名表字段 |
   | ABV | 酒精度 |
   | 酒款简介 | 厂商填写，评审可见 |
   | 允许公开的额外字段 | 由报名表字段配置控制 |

   评审不可见信息：

   | 不展示 | 原因 |
   | --- | --- |
   | 酒名 | 避免识别酒款 |
   | 厂商信息 | 保持匿名评审 |
   | 联系人信息 | 与评审无关 |
   | 其他可识别厂商的信息 | 保持评审公平 |

4. **普通评审独立评分**

   跨界评审和专业评审按自己的角色加载评分表，填写各维度分数和备注。

   - 系统实时汇总总分。
   - 单项分数不能超过该维度满分。
   - 所有分数必须为整数。
   - 默认总分口径为 50 分。
   - 专业评审 5 个维度备注合计不少于 30 字。
   - 跨界评审各维度备注合计不少于 50 字。
   - 专业评审的“整体印象备注”承担原总评语作用，不再额外重复设置一个总评语字段。

5. **普通评审提交与修改**

   普通评审提交评分后，在桌长汇总前可修改。建议系统以“该酒款尚未生成桌长共识分”作为可修改边界；一旦桌长提交该酒款汇总，普通评审不可再修改。

#### 阶段 5：第一轮桌长汇总与晋级

1. **进入本轮汇总**

   同桌评审完成本桌本轮所有酒款评分后，桌长进入汇总视图。第一轮按评分制处理，每款酒都有普通评审原始评分和桌长共识分。

2. **查看原始评分**

   桌长可查看同桌所有评审对每款酒提交的完整原始评分表、各维度分数、总分和文字反馈。

3. **填写共识分**

   桌长基于同桌讨论和原始评分，为每款酒填写共识分。共识分是讨论后的独立最终分，不自动等于平均分，也不绑定某一位评审的个人分。

4. **填写综合评语**

   桌长为每款酒填写综合评语。该评语会在结果发布后展示给厂商。

5. **勾选晋级酒款**

   本桌本轮全部酒款评完后，桌长从该批次中勾选晋级酒款。系统按后台配置校验晋级数量，例如该桌配置为 3 款，则桌长必须选且只能选 3 款。

6. **提交桌长汇总**

   桌长提交共识分、综合评语和晋级结果。结果锁定前，桌长应可继续调整共识分、综合评语和晋级标记。

7. **汇总锁定**

   具体锁定动作待确认。建议候选方案包括：主办方确认后锁定、本轮关闭后锁定、结果发布后锁定。锁定后普通评审和桌长均不可再修改评分数据。

#### 阶段 6：后续轮评审与奖牌排序

1. **创建后续轮评审桌**

   比赛通常至少 2 轮，可能有 3 轮。第一轮完成后，主办方根据晋级结果在后台创建第二轮或第三轮评审桌，例如 2A、2B、3A。

2. **分配后续轮评审和酒款**

   后续轮的评审人员和酒款可提前配置，也需要支持现场临时调整。主办方可从晋级酒款池中筛选、搜索、拖动或勾选酒款到后续轮评审桌。

3. **后续轮选择与排序**

   第二轮及之后以淘汰和排序为主，通常不再要求普通评审逐项打分和写备注。桌长或指定评审组从本轮酒款中选择奖牌候选，并排序出前三名。

4. **确定组别金银铜**

   每个投递组别最终确认金奖、银奖、铜奖三款酒。具体由后续轮选择排序结果形成，主办方可在后台确认。

5. **确定全场总冠军**

   最终轮把各投递组别金奖放入总冠军候选池，评审团从中选出 1 款全场总冠军。

#### 阶段 7：后台监控与数据校验

1. **查看评审进度**

   主办方后台按组别、轮次、评审桌查看评审进度。“已评审”的定义按桌长已汇总计算。

2. **查看晋级情况**

   主办方查看各轮次、各评审桌勾选的晋级酒款。

3. **查看评审工作量**

   主办方查看每位评审评了多少酒、每位评审对每款酒的平均耗时、每款酒和每位评审的备注字数是否满足要求。

   现场看板应支持投屏或公开展示，让评审看到整体进度、平均耗时和平均字数，方便现场调整节奏。

4. **导出评分数据**

   主办方可导出完整评分数据，包含每位评审原始评分、文字反馈、桌长共识分、桌长综合评语、晋级标记和奖项结果。

#### 阶段 8：奖项确认与结果发布

1. **形成组别候选**

   多轮评审晋级后，同一投递组别内的晋级酒款形成组别获奖候选。

2. **确认组别奖项**

   默认规则为每个投递组别确认金奖、银奖、铜奖。主办方 / 评审团结合后续轮选择排序结果确认获奖酒款。

   注意：分数只在同评审桌、同组别、同一评分口径下具备较强参考意义。如果候选酒款来自不同评审桌，分数只能作为参考，避免直接按跨桌分数做全场排名。

3. **确认全场总冠军**

   默认口径为从各组别金奖作品中综合评定 1 款全场总冠军。具体评选流程仍需业务确认。

4. **发布结果**

   主办方发布结果后，酒款状态更新为“结果已出”。发布后厂商可在 Portal 查看自己提交酒款的评审反馈和获奖信息。

5. **厂商查看反馈**

   厂商只能查看自己提交酒款的结果和反馈。为保证公平、公正、公开，结果页应展示：

   | 内容 | 说明 |
   | --- | --- |
   | 每位评审原始评分表 | 包含各维度分数、总分和文字反馈 |
   | 桌长共识分 | 桌长汇总后的最终分 |
   | 桌长综合评语 | 桌长填写的总结反馈 |
   | 晋级情况 | 该酒款是否在对应轮次晋级 |
   | 奖项结果 | 是否获奖、奖项名称、是否全场总冠军 |
   | 组别参赛作品总数 | 用于理解该酒款所在组别规模 |

6. **获奖后处理**

   获奖厂商可下载奖状，并填写奖牌 / 奖状收件地址。

### 二、关键业务口径

| 事项 | 当前口径 |
| --- | --- |
| 每场比赛配置 | 默认模板 + 单场覆盖 |
| 报名表下拉选项 | 投递组别按比赛配置；基础风格使用全局标准风格库，必填、单选、可搜索 |
| 评分表 | 专业评审固定 5 项；跨界评审按比赛配置；默认总分 50 |
| 评分输入 | 分数必须为整数，每个维度下方有备注框，字数按所有备注合计校验 |
| 评审匿名信息 | 展示匿名编码、组别、风格、ABV、简介和允许公开的额外字段 |
| 隐藏信息 | 不展示酒名、厂商、联系人和其他可识别厂商的信息 |
| 普通评审修改 | 桌长汇总前可修改 |
| 已评审定义 | 桌长已汇总 |
| 桌长汇总 | 查看原始评分，填写共识分、综合评语，勾选晋级 |
| 晋级 | 发生在轮次/评审桌级别，数量按每桌配置，桌长必须选且只能选 N 款 |
| 后续轮 | 第二轮及之后以选择和排序为主，通常不再打分和写备注 |
| 编号兜底 | 二维码下方显示短编号，扫码失败时可手动输入 |
| 现场看板 | 支持大屏/公开展示整体进度、平均耗时、备注字数 |
| 分数排名 | 避免跨桌直接排名，跨桌候选由主办方 / 评审团综合确认 |
| 厂商结果可见 | 展示每位评审原始评分表、备注、桌长总结、共识分和奖项 |

### 三、待确认事项

以下事项从当前需求文档无法推导出明确答案，需要业务方、主办方或评审专家进一步确认。

| 编号 | 待确认事项 | 影响范围 |
| --- | --- | --- |
| 1 | 本次比赛是否必须上线厂商报名和微信支付 | 决定厂商 Portal、支付、二维码下载是否进入 P0 |
| 2 | 跨界评审、媒体评审、大众评审最终名称 | 页面文案、角色展示、评分表命名 |
| 3 | 第一版报名表默认模板字段 | 厂商报名页、后台配置页、数据库字段 |
| 4 | 每场比赛投递组别完整列表 | 报名表下拉选项、结果分组、奖项设置 |
| 5 | 标准基础风格库采用哪一版指南、是否需要导入约 200 个风格 | 报名表、评审扫码页、后台配置 |
| 6 | 哪些报名表额外字段对评审可见 | 匿名展示边界、评分参考信息 |
| 7 | 跨界评审第一版评分维度、每项满分、反馈字段 | H5 评分页、后端校验、ScoreConfig |
| 8 | 专业评审第一版评分维度、每项满分、反馈字段 | H5 评分页、后端校验、ScoreConfig |
| 9 | 桌长综合评语是否需要至少 30 字 | 前端校验、后端校验、厂商结果展示 |
| 10 | 每张评审桌的默认人数规则 | 评审分配页、桌长唯一性校验 |
| 11 | 每一轮每张桌默认评多少款酒、晋级几款 | 轮次配置、桌长汇总页、晋级规则 |
| 12 | 后续轮由谁操作选择和排序：桌长、全体评审，还是主办方代录 | 后续轮 H5、后台录入、权限边界 |
| 13 | 桌长提交后哪个动作触发最终锁定 | 修改权限、审计记录、现场纠错流程 |
| 14 | 主办方是否可在后台修改桌长共识分和晋级结果 | 权限边界、审计记录、数据公信力 |
| 15 | 同组别跨桌候选如何最终比较 | 金银铜确认规则、结果公平性 |
| 16 | 评审端编号是否需要和厂商端编号做双编号映射 | 二维码、手动输入、匿名公平性 |
| 17 | 奖项规则是否每场比赛都允许覆盖默认模板 | 奖项配置页、结果发布流程 |
| 18 | 厂商结果页是否展示评审姓名和角色 | 隐私口径、公开透明程度 |
| 19 | 厂商可见反馈的开放时间 | 结果发布状态、Portal 权限控制 |
| 20 | 支付主体、退款规则、发票信息 | 微信支付、退款、财务对账 |
| 21 | 是否需要关键状态短信通知 | 短信模板、运营成本、触发规则 |

**轮次口径：第一轮是评分制，普通评审打分和写备注，桌长给共识分并选择晋级；第二轮及之后主要做选择和排序，用于确定组别金银铜；最终轮从各组别金奖中选出全场总冠军。**
