### 一、啤酒大赛完整流程

本文档按当前需求分析报告整理比赛从赛前配置到赛后发布的业务流程。涉及评分表、Flight、桌长锁定、奖项规则的部分，以下方“待确认事项”为准；在业务最终确认前，系统设计应保留配置和调整空间。

#### 阶段 1：赛前筹备与比赛配置

1. **创建比赛基础信息**

   主办方在后台创建比赛，设置比赛名称、比赛日期、报名截止时间、报名费金额、比赛状态等基础信息。

2. **从默认模板生成本场配置**

   系统内置报名表模板、评分表模板、奖项规则模板。创建比赛时复制默认模板到当前比赛，后续主办方可按本场比赛调整，不影响其他比赛和历史比赛。

3. **配置报名表**

   主办方配置本场比赛报名表字段，包括酒名、投递组别、基础风格、ABV、酒款简介和额外字段。

   - 投递组别是报名表下拉选项，每场比赛单独维护。
   - 基础风格可使用默认风格库，也可按比赛启用或补充选项。
   - 额外字段可配置字段类型、下拉选项、是否必填、展示顺序、是否在评审扫码页展示。

4. **配置评分表**

   主办方按角色配置评分表：跨界评审、专业评审、桌长分别维护。

   - 每个评分维度需要配置名称、说明、满分、排序。
   - 默认总分口径为 50 分，系统保存配置时应校验各维度满分之和。
   - 文字反馈字段可配置；如有多个指定反馈栏目，需要明确字数校验口径。

5. **配置评审账号与评审桌**

   主办方维护评审账号信息，包括姓名、手机号、微信号、资质信息等。每场比赛单独创建评审桌，例如 A 桌、B 桌。

   - 评审角色与单场比赛绑定，同一评审在不同比赛可担任不同角色。
   - 每张评审桌可配置不同人数。
   - 每张评审桌必须有且只有 1 名桌长。
   - 每桌可配置 N 名跨界评审和 N 名专业评审，具体人数按比赛实际情况设置。

6. **配置 Flight/批次**

   如果本场比赛启用 Flight 管理，主办方按评审桌配置每个 Flight 的酒款范围和晋级数量。

   - 一个 Flight 通常约 10 款酒。
   - 晋级数量可按业务规则配置，例如 10 进 2 或 2-4 款。
   - 酒款与评审桌的匹配由主办方线下安排，系统不自动分配。

7. **配置奖项规则**

   系统默认奖项规则为：每个投递组别设置金奖、银奖、铜奖；每场比赛设置 1 个全场总冠军。主办方可按比赛覆盖默认规则，调整奖项名称、数量、排序和启用状态。

#### 阶段 2：厂商报名参赛

1. **厂商注册/登录**

   厂商联系人使用手机号注册并短信验证码登录，填写联系人姓名、手机号、微信号和厂牌信息。

2. **选择比赛并提交酒款**

   厂商选择目标比赛，按本场比赛报名表填写酒款信息。同一厂商账号可参加多场比赛，也可在同一比赛提交多款参赛酒。

3. **生成 UUID**

   酒款提交后，系统为该酒款生成唯一 UUID，作为匿名编码。评审现场通过二维码扫描 UUID，不直接看到酒名和厂商信息。

4. **支付报名费**

   厂商按本场比赛配置的报名费付款。支付成功后，酒款状态更新为“已付款，报名成功”。支付失败或超时可重新发起。

5. **查看状态和二维码**

   厂商可在 Portal 查看自己提交酒款的状态和 UUID。支付成功后，厂商可下载二维码用于确认和打印。

#### 阶段 3：酒款入库与现场准备

1. **主办方接收酒款**

   主办方线下接收参赛酒款，并在后台核对报名信息、UUID 和酒款状态。

2. **更新入库状态**

   酒款收到后，主办方在后台将状态更新为“已入库”。如报名取消或退款，状态更新为“已取消”。

3. **准备二维码**

   主办方可在后台查看、下载、批量导出二维码。比赛当天，主办方或现场执行人员将二维码贴到对应酒瓶上。

4. **核对评审桌和 Flight**

   现场开始前，主办方核对评审桌、评审人员、酒款批次和二维码，确保评审扫码后进入正确的匿名酒款信息页。

#### 阶段 4：现场评审

1. **评审登录**

   评审使用手机号 + 验证码登录。登录后从自己被分配的比赛列表中选择本场比赛；如果只有一场分配比赛，可自动进入。

2. **查看角色和评审桌**

   系统展示评审在当前比赛中的角色、评审桌和相关提示。角色包括跨界评审、专业评审、桌长。

3. **扫码获取匿名酒款信息**

   评审扫描瓶身二维码，系统根据 UUID 展示匿名酒款信息。

   评审可见信息：

   | 可展示 | 说明 |
   | --- | --- |
   | UUID | 匿名编码 |
   | 投递组别 | 当前比赛配置的组别 |
   | 基础风格 | 当前报名表字段 |
   | ABV | 酒精度 |
   | 酒款简介 | 厂商填写，评审可见 |
   | 允许公开的额外字段 | 由报名表字段配置控制 |

   评审不可见信息：

   | 不展示 | 原因 |
   | --- | --- |
   | 酒名 | 避免识别酒款 |
   | 厂商信息 | 保持匿名评审 |
   | 联系人信息 | 与评审无关 |
   | 其他可识别厂商的信息 | 保持评审公平 |

4. **普通评审独立评分**

   跨界评审和专业评审按自己的角色加载评分表，填写各维度分数和文字反馈。

   - 系统实时汇总总分。
   - 单项分数不能超过该维度满分。
   - 默认总分口径为 50 分。
   - 跨界评审文字反馈不少于 50 字，专业评审不少于 30 字；如后续拆分多个反馈栏目，需按最终规则校验。

5. **普通评审提交与修改**

   普通评审提交评分后，在桌长汇总前可修改。建议系统以“该酒款尚未生成桌长最终评分”作为可修改边界；一旦桌长提交该酒款汇总，普通评审不可再修改。

#### 阶段 5：桌长汇总与 Flight 晋级

1. **进入 Flight 汇总**

   同桌评审完成一个 Flight 的所有酒款评分后，桌长进入该 Flight 的汇总视图。一个 Flight 通常约 10 款酒，具体数量按比赛配置或现场安排。

2. **查看原始评分**

   桌长可查看同桌所有评审对每款酒提交的完整原始评分表、各维度分数、总分和文字反馈。

3. **填写最终评分**

   桌长基于同桌讨论和原始评分，为每款酒填写调整后的最终评分。当前暂按“桌长填写最终评分”处理；最终评分是一个总分，还是逐维度调整后自动汇总，需要业务进一步确认。

4. **填写综合评语**

   桌长为每款酒填写综合评语。该评语会在结果发布后展示给厂商。

5. **勾选晋级酒款**

   一个 Flight 全部酒款评完后，桌长从该批次中勾选晋级酒款，例如 10 进 2，或按配置选择 2-4 款。系统应按本场规则限制或提示晋级数量。

6. **提交桌长汇总**

   桌长提交最终评分、综合评语和晋级结果。结果锁定前，桌长应可继续调整最终评分、综合评语和晋级标记。

7. **汇总锁定**

   具体锁定动作待确认。建议候选方案包括：主办方确认后锁定、Flight 关闭后锁定、结果发布后锁定。锁定后普通评审和桌长均不可再修改评分数据。

#### 阶段 6：后台监控与数据校验

1. **查看评审进度**

   主办方后台按组别、评审桌、Flight 查看评审进度。“已评审”的定义按桌长已汇总计算。

2. **查看晋级情况**

   主办方查看各评审桌、各 Flight 勾选的晋级酒款。

3. **查看评审工作量**

   主办方查看每位评审评了多少酒、每款酒评语字数是否满足要求。

4. **导出评分数据**

   主办方可导出完整评分数据，包含每位评审原始评分、文字反馈、桌长最终评分、桌长综合评语、晋级标记和奖项结果。

#### 阶段 7：奖项确认与结果发布

1. **形成组别候选**

   多轮 Flight 晋级后，同一投递组别内的晋级酒款形成组别获奖候选。

2. **确认组别奖项**

   默认规则为每个投递组别确认金奖、银奖、铜奖。主办方 / 评审团结合晋级情况和桌长最终评分确认获奖酒款。

   注意：分数只在同评审桌、同组别、同一评分口径下具备较强参考意义。如果候选酒款来自不同评审桌，分数只能作为参考，避免直接按跨桌分数做全场排名。

3. **确认全场总冠军**

   默认口径为从各组别金奖作品中综合评定 1 款全场总冠军。具体评选流程仍需业务确认。

4. **发布结果**

   主办方发布结果后，酒款状态更新为“结果已出”。发布后厂商可在 Portal 查看自己提交酒款的评审反馈和获奖信息。

5. **厂商查看反馈**

   厂商只能查看自己提交酒款的结果和反馈。为保证公平、公正、公开，结果页应展示：

   | 内容 | 说明 |
   | --- | --- |
   | 每位评审原始评分表 | 包含各维度分数、总分和文字反馈 |
   | 桌长最终评分 | 桌长汇总后的最终评分 |
   | 桌长综合评语 | 桌长填写的总结反馈 |
   | 晋级情况 | 该酒款是否在对应 Flight 或轮次晋级 |
   | 奖项结果 | 是否获奖、奖项名称、是否全场总冠军 |
   | 组别参赛作品总数 | 用于理解该酒款所在组别规模 |

6. **获奖后处理**

   获奖厂商可下载奖状，并填写奖牌 / 奖状收件地址。

### 二、关键业务口径

| 事项 | 当前口径 |
| --- | --- |
| 每场比赛配置 | 默认模板 + 单场覆盖 |
| 报名表下拉选项 | 投递组别必须按比赛配置，基础风格建议支持按比赛启用或扩展 |
| 评分表 | 按比赛、角色配置，默认总分 50 |
| 评审匿名信息 | 展示 UUID、组别、风格、ABV、简介和允许公开的额外字段 |
| 隐藏信息 | 不展示酒名、厂商、联系人和其他可识别厂商的信息 |
| 普通评审修改 | 桌长汇总前可修改 |
| 已评审定义 | 桌长已汇总 |
| 桌长汇总 | 查看原始评分，填写最终评分、综合评语，勾选晋级 |
| 晋级 | 发生在 Flight 级别，数量按比赛规则配置 |
| 分数排名 | 避免跨桌直接排名，跨桌候选由主办方 / 评审团综合确认 |
| 厂商结果可见 | 展示每位评审原始评分表、评语、桌长总结、最终评分和奖项 |

### 三、待确认事项

以下事项从当前需求文档无法推导出明确答案，需要业务方、主办方或评审专家进一步确认。

| 编号 | 待确认事项 | 影响范围 |
| --- | --- | --- |
| 1 | 本次比赛是否必须上线厂商报名和微信支付 | 决定厂商 Portal、支付、二维码下载是否进入 P0 |
| 2 | 跨界评审、媒体评审、大众评审最终名称 | 页面文案、角色展示、评分表命名 |
| 3 | 第一版报名表默认模板字段 | 厂商报名页、后台配置页、数据库字段 |
| 4 | 每场比赛投递组别完整列表 | 报名表下拉选项、结果分组、奖项设置 |
| 5 | 基础风格是否每场比赛独立配置 | 报名表、评审扫码页、后台配置 |
| 6 | 哪些报名表额外字段对评审可见 | 匿名展示边界、评分参考信息 |
| 7 | 跨界评审第一版评分维度、每项满分、反馈字段 | H5 评分页、后端校验、ScoreConfig |
| 8 | 专业评审第一版评分维度、每项满分、反馈字段 | H5 评分页、后端校验、ScoreConfig |
| 9 | 评语字数按单个字段校验，还是多个反馈栏目合计校验 | 前端校验、后端校验、厂商结果展示 |
| 10 | 每张评审桌的默认人数规则 | 评审分配页、桌长唯一性校验 |
| 11 | Flight 的正式定义：每批多少款、谁创建、酒款如何进入 Flight | Flight 配置、桌长汇总页、晋级规则 |
|      |                                                            |                                              |
| 13 | 桌长最终评分是只填总分，还是逐维度调整后汇总 | 桌长页面、评分数据模型、厂商结果页 |
| 14 | 桌长提交后哪个动作触发最终锁定 | 修改权限、审计记录、现场纠错流程 |
| 15 | 主办方是否可在后台修改桌长最终结果 | 权限边界、审计记录、数据公信力 |
| 16 | 同组别跨桌候选如何最终比较 | 金银铜确认规则、结果公平性 |
|      |                                                            |                                              |
| 18 | 全场总冠军是否必须从组别金奖中产生 | 奖项规则、后台结果确认流程 |
| 19 | 奖项规则是否每场比赛都允许覆盖默认模板 | 奖项配置页、结果发布流程 |
| 20 | 厂商结果页是否展示评审姓名和角色 | 隐私口径、公开透明程度 |
| 21 | 厂商可见反馈的开放时间 | 结果发布状态、Portal 权限控制 |
| 22 | 支付主体、退款规则、发票信息 | 微信支付、退款、财务对账 |
| 23 | 是否需要关键状态短信通知 | 短信模板、运营成本、触发规则 |

**Flight 是“同组别里的一小批酒”。先在每个 Flight 里筛出好酒，再把同组别的晋级酒汇总成候选池。候选池后面要不要重新喝一轮，现在还需要业务确认。**
