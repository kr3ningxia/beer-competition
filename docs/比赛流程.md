### 一、啤酒大赛完整流程

本文档按当前系统实现和最新需求口径整理比赛从赛前配置到赛后发布的业务流程。流程中已实现的系统动作直接按当前系统描述；仍需业务确认或后续开发的内容放在“待确认事项”。

#### 阶段 1：赛前筹备与比赛配置

1. **创建比赛基础信息**

   主办方在后台创建比赛，初始状态为 `DRAFT`。创建时需要填写：

   | 配置项 | 说明 |
   | --- | --- |
   | 比赛名称、比赛日期 | 厂商端公开展示 |
   | 报名开始时间、报名截止时间 | 控制报名窗口 |
   | 普通报名费、早鸟价、早鸟截止时间 | 控制厂家端当前应付金额 |
   | 赛事简介 | 厂商端赛事详情展示 |
   | 基础风格库 | 例如 BJCP 2021 中文库、英文库或主办方自定义库 |
   | 送样方式 | 快递、现场送样或两者都支持 |
   | 收样时间、数量要求、收件信息、送样说明 | 厂商付款后或按可见规则展示 |
   | 物流可见规则 | 公开、登录后可见、付款确认后可见 |

2. **配置报名组别和报名字段**

   主办方配置本场比赛的投递组别、基础风格库和额外报名字段。

   - 投递组别是报名表下拉选项，每场比赛单独维护，例如浅色拉格、深色拉格、创意拉格。
   - 基础风格来自当前比赛绑定的风格库，厂商报名时必填、单选、可搜索。
   - 酒款简介不作为固定字段；特殊原料、工艺、桶陈、增味说明等通过额外字段配置。
   - 额外字段可配置字段类型、选项、是否必填、展示顺序、是否在评审扫码页展示。

3. **配置评分表**

   主办方按角色配置评分表：跨界评审、专业评审、桌长分别维护。

   - 专业评审第一版固定 5 个维度：香气 12 分、外观 3 分、味道 20 分、口感 5 分、整体印象 10 分，总分 50 分。
   - 跨界评审每场比赛可配置 2-3 个维度，维度名称、说明、满分和排序均可调整，总分 50 分。
   - 每个评分维度下方都有备注框。
   - 评分必须为整数，单项分数不能超过维度满分。
   - 专业评审备注合计不少于 30 字，跨界评审备注合计不少于 50 字；具体字数以后台评分表配置为准。
   - 桌长综合评语的最小字数也由桌长评分配置控制。

4. **配置评审账号与基础评审桌**

   主办方维护评审账号信息，包括姓名、手机号、微信号、资质信息和审核状态。每场比赛单独配置基础评审桌，例如 A 桌、B 桌。

   - 评审角色与单场比赛绑定，同一评审在不同比赛可担任不同角色。
   - 每张评审桌可配置不同人数。
   - 每张评审桌必须有且只有 1 名桌长。
   - 每桌可配置 N 名跨界评审和 N 名专业评审，具体人数按比赛实际情况设置。

5. **开放报名**

   草稿比赛配置通过检查后，主办方点击开放报名，比赛状态进入 `REGISTRATION_OPEN`。厂商端公开赛事列表和赛事详情会展示该比赛，并开放提交酒款入口。

6. **关闭报名并进入评审准备**

   报名截止或主办方手动关闭后，比赛状态进入 `REGISTRATION_CLOSED`。主办方完成报名核对、收样准备和评审编排后，将比赛推进到 `JUDGING_PREP`。

#### 阶段 2：厂商报名参赛

1. **浏览赛事**

   厂商未登录时可浏览公开首页、赛事列表、赛事详情和公开结果。赛事详情展示报名时间、费用、早鸟价、投递组别、基础风格、送样要求和结果查看口径。

2. **厂商登录与资料维护**

   厂商联系人使用手机号和短信验证码登录。登录后可维护联系人姓名、手机号、微信号和厂牌信息。微信号是否强制必填需业务最终确认。

3. **选择比赛并提交酒款**

   厂商只能向 `REGISTRATION_OPEN` 且未超过报名截止时间的比赛提交酒款。同一厂商账号可参加多场比赛，也可在同一比赛提交多款参赛酒。

   报名表字段包括：

   | 字段 | 规则 |
   | --- | --- |
   | 酒名 | 必填，仅后台和厂商可见 |
   | 投递组别 | 必填，从本场比赛组别中选择 |
   | 基础风格 | 必填，从本场比赛风格库中选择 |
   | ABV | 必填，0 到 99.99，最多两位小数 |
   | 额外字段 | 按后台字段配置校验 |

4. **生成匿名编号和现场标签**

   酒款提交后，系统生成：

   | 编号 | 用途 |
   | --- | --- |
   | 系统匿名编号 | 系统内酒款标识 |
   | 标签编码 | 后台核对和内部追踪 |
   | 短编号 | 现场参赛标签展示，扫码失败时可手动输入 |
   | 扫码令牌 | 二维码跳转和评审端解析 |

   现场参赛标签展示二维码、参赛编号（短编号）和组别。编号应保持随机性，避免从编号识别厂商或酒款。

5. **支付报名费**

   厂商按当前应付金额付款。当前系统支持模拟支付和后台手动确认付款；真实微信支付、回调、退款和对账仍需接入。

   支付确认后：

   - 酒款状态从 `PENDING_PAYMENT` 变为 `REGISTERED`。
   - 付款状态变为 `PAID`。
   - 厂商端开放现场参赛标签下载。
   - 若物流可见规则为付款后可见，则展示完整收件信息。

6. **下载标签并提交送样信息**

   厂商付款后可下载现场参赛标签。当前标签版式包含二维码、参赛编号（短编号）和组别；风格、ABV 和评审可见额外字段在评审扫码进入匿名酒款页后展示，不印在参赛标签上。

   厂商按赛事配置提交送样信息：

   | 送样方式 | 填写要求 |
   | --- | --- |
   | 快递寄送 | 快递公司、快递单号必填，可填写备注 |
   | 现场送样 | 可填写备注 |

7. **查看参赛状态**

   厂商可在“我的参赛”和“我的酒款”查看每款酒状态。

   | 状态线 | 编码 | 含义 |
   | --- | --- | --- |
   | 酒款状态 | `PENDING_PAYMENT` | 待支付 |
   | 酒款状态 | `REGISTERED` | 报名成功，标签可用 |
   | 酒款状态 | `STORED` | 主办方确认入库 |
   | 酒款状态 | `CANCELED` | 报名取消 |
   | 酒款状态 | `RESULT_PUBLISHED` | 结果已发布 |
   | 付款状态 | `UNPAID`、`PAID`、`CANCELED`、`REFUNDED` | 付款处理 |
   | 送样状态 | `NOT_SUBMITTED`、`SUBMITTED`、`RECEIVED` | 送样和收样处理 |

#### 阶段 3：酒款入库与现场准备

1. **主办方核对报名酒款**

   主办方在后台酒款管理页按比赛、报名状态、付款状态、送样状态、组别、是否分桌和关键词筛选酒款，核对酒名、厂商、组别、风格、ABV、额外字段、标签编码和短编号。

2. **修正酒款报名信息**

   厂商提交酒款后，如主办方发现酒名、投递组别、基础风格、ABV 或额外字段有误，可在后台手动编辑。

   - 修改原因必填。
   - 修改投递组别时，如果酒款已分桌或进入后续轮候选，需要重新核对分桌、晋级和奖项归属。
   - 修改基础风格、ABV 或评审可见额外字段时，如果评审已开始，需要注意评审端展示口径变化。
   - 匿名编号、标签编码和短编号默认不随报名信息修改而变化。
   - 系统记录修改日志和分桌轨迹。

3. **确认付款、入库或取消**

   主办方可在后台：

   | 动作 | 影响 |
   | --- | --- |
   | 确认付款 | 付款状态变为 `PAID`，酒款状态变为 `REGISTERED` |
   | 确认入库 | 送样状态变为 `RECEIVED`，酒款状态变为 `STORED` |
   | 取消报名 | 酒款状态变为 `CANCELED`，不进入分桌池 |

4. **准备现场标签**

   厂商端已支持单款参赛标签下载。主办方后台可查看标签编码和短编号，用于现场核对；后台导出中心已支持按筛选范围批量导出 A4 四联参赛标签 PDF，并保留图片文件包用于单张补打。

5. **核对评审桌和轮次**

   现场开始前，主办方核对评审账号、评审桌、轮次、酒款分配、标签编码和短编号，确保评审扫码后进入正确的匿名酒款页面。

#### 阶段 4：评审账号准备

1. **评审注册或后台维护**

   评审可在 H5 注册账号，并补充个人资料；主办方也可在后台维护评审资料。

2. **审核评审资料**

   主办方审核评审账号，启用后才能加入比赛编排。已分配比赛的评审不能直接更正手机号，需要按后台手机号更正规则处理。

3. **发布评审任务**

   比赛进入评审准备后，主办方创建轮次并发布。评审登录后从自己的任务列表进入对应比赛、轮次和评审桌。

#### 阶段 5：第一轮评分

1. **评审登录**

   评审使用手机号和验证码登录。若资料未完善或仍在审核中，进入资料或审核状态页面；启用并被编排后进入任务列表。

2. **查看任务**

   评审端展示当前比赛、轮次、评审桌、角色、酒款数量、已完成数量和当前任务类型。

3. **扫码或输入短编号**

   评审扫描瓶身二维码或输入短编号进入匿名酒款页。

   评审可见：

   | 可展示 | 说明 |
   | --- | --- |
   | 标签编码或短编号 | 用于现场核对 |
   | 投递组别 | 当前比赛配置的组别 |
   | 基础风格 | 当前报名表字段 |
   | ABV | 酒精度 |
   | 允许公开的额外字段 | 由报名字段配置控制 |
   | 比赛、轮次、评审桌 | 用于确认当前任务 |

   评审不可见：

   | 不展示 | 原因 |
   | --- | --- |
   | 酒名 | 避免识别酒款 |
   | 厂商信息 | 保持匿名评审 |
   | 联系人信息 | 与评审无关 |
   | 其他可识别厂商的信息 | 保持评审公平 |

4. **普通评审独立评分**

   跨界评审和专业评审按自己的角色加载评分表，填写各维度分数和备注。

   - 系统实时汇总总分。
   - 单项分数不能超过维度满分。
   - 所有分数必须为整数。
   - 默认总分口径为 50 分。
   - 备注合计字数按当前评分表配置校验。
   - 在桌长汇总或本桌结果确认前，个人评分可查看和修改。

5. **桌长个人评分**

   桌长也需要先完成自己的个人评分。当前系统中，桌长个人评分使用专业评审评分表；该记录参与本桌个人评分进度，但不等同于桌长共识分。

#### 阶段 6：第一轮桌长汇总与同桌确认

1. **进入桌长工作台**

   同桌评审完成评分后，桌长进入本桌工作台。桌长可查看每款酒的同桌评分提交情况。

2. **查看原始评分**

   桌长可查看同桌所有评审对每款酒提交的完整原始评分表、维度分数、总分和文字反馈，包含桌长自己的个人评分。

3. **填写共识分和综合评语**

   桌长基于同桌讨论和原始评分，为每款酒填写共识分和综合评语。

   - 共识分是讨论后的独立最终分。
   - 共识分不自动等于平均分，也不绑定某一位评审的个人分。
   - 综合评语会在结果发布后展示给厂商。

4. **勾选晋级酒款**

   桌长按本桌目标数量勾选晋级酒款。系统按后台配置校验，例如该桌需要晋级 3 款，则必须选且只能选 3 款。

5. **同桌确认**

   桌长完成本桌结果后，同桌评审进入确认页，核对共识分、综合评语和晋级名单。

   - 普通评审确认后，系统记录确认状态。
   - 本桌结果进入主办方确认前，需要同桌确认达到要求。
   - 如现场有特殊情况，主办方可在后台覆盖确认，覆盖原因需要记录。

6. **本桌结果进入主办方确认**

   同桌确认完成后，本桌结果进入主办方确认。本桌结果提交后，普通评分进入锁定边界。

7. **主办方锁定轮次**

   主办方核对各桌结果后，可锁定轮次。锁定后轮次结果进入后续轮候选池，普通评审和桌长原则上不可继续修改本轮结果。

#### 阶段 7：后续轮选择与排序

1. **创建后续轮**

   第一轮完成并锁定后，主办方根据晋级结果创建第二轮或第三轮。后续轮类型为 `RANKING`。

2. **同步候选酒款**

   主办方从上一轮晋级结果同步候选酒款，也可按现场需要调整后续轮评审桌和候选池。

3. **配置目标模式**

   后续轮支持不同目标模式：

   | 目标模式 | 用途 |
   | --- | --- |
   | `TOP_N` | 选出前 N 名 |
   | `MEDALS` | 排出金奖、银奖、铜奖 |
   | `CHAMPION` | 选出总冠军 |

4. **评审排序**

   后续轮通常不再逐项打分和写备注。

   - 桌长提交正式排序结果。
   - 参与评审可查看候选并保存参考排序草稿。
   - 主办方可根据排序结果确认轮次结果。

5. **锁定后续轮**

   后续轮排序完成后，主办方锁定轮次。锁定后的排序结果可用于生成奖项草稿。

#### 阶段 8：后台监控与数据校验

1. **查看评审进度**

   主办方后台按比赛、轮次、评审桌查看酒款、评分、桌长汇总、同桌确认和排序进度。

2. **查看晋级和排序结果**

   主办方查看每轮每桌晋级酒款、后续轮排序结果和候选池变化。

3. **查看反馈质量**

   系统应持续支持主办方查看每位评审评分数量、备注字数、平均耗时等。当前已有进度基础，平均耗时和字数质量的大屏指标需进一步验收。

4. **现场看板**

   后台已有现场看板入口。现场投屏版本应优先展示整体进度、各桌进度、待处理事项和反馈质量，避免展示跨桌分数排名造成误导。

5. **导出评分数据**

   主办方可导出完整评分数据，包含酒款信息、评审原始评分、文字反馈、桌长共识分、桌长综合评语、轮次结果和奖项结果。

#### 阶段 9：奖项确认与结果发布

1. **生成奖项草稿**

   比赛进入 `RESULT_CONFIRMING` 后，主办方可基于锁定的排序轮生成奖项草稿。

   当前默认规则：

   | 奖项 | 来源 |
   | --- | --- |
   | 组别金奖、银奖、铜奖 | `MEDALS` 目标模式排序结果 |
   | 全场总冠军 | `CHAMPION` 目标模式排序结果，或从各组别金奖中确认 |

2. **确认奖项**

   主办方核对奖项草稿后确认奖项。分数只在同评审桌、同组别、同一评分口径下具备较强参考意义；跨桌候选应通过后续轮排序和主办方确认决定。

3. **上传证书**

   主办方可为已确认或已发布的奖项上传 PDF 证书。证书上传后，后台和获奖厂商可下载。

4. **发布结果**

   奖项确认完成后，主办方发布结果。

   发布后：

   - 比赛状态进入 `PUBLISHED`。
   - 获奖结果状态进入发布口径。
   - 厂商端开放个人结果查询。
   - 厂商端可下载自己的获奖证书。

5. **厂商查看反馈**

   厂商只能查看自己提交酒款的结果和反馈。结果页应展示：

   | 内容 | 说明 |
   | --- | --- |
   | 每位评审原始评分表 | 包含维度分数、总分和文字反馈 |
   | 桌长共识分 | 桌长汇总后的最终分 |
   | 桌长综合评语 | 桌长填写的总结反馈 |
   | 轮次结果 | 每轮晋级、排序或淘汰情况 |
   | 奖项结果 | 是否获奖、奖项名称、是否总冠军 |
   | 组别参赛作品总数 | 用于理解该酒款所在组别规模 |
   | 证书下载 | 获奖且主办方上传证书后开放 |

6. **获奖后处理**

   当前系统支持证书下载。奖牌/奖状收件地址尚未实现，如业务需要邮寄奖牌或实体奖状，应补充厂商端地址填写和后台管理。

### 二、关键业务口径

| 事项 | 当前口径 |
| --- | --- |
| 比赛状态 | `DRAFT`、`REGISTRATION_OPEN`、`REGISTRATION_CLOSED`、`JUDGING_PREP`、`JUDGING`、`RESULT_CONFIRMING`、`PUBLISHED`、`ARCHIVED` |
| 厂商报名窗口 | 只有报名开放且未截止的比赛可提交酒款 |
| 报名费用 | 支持普通报名费和早鸟价；真实微信支付待接入 |
| 送样信息 | 支持快递、现场送样或两者都支持；完整收件信息按可见规则展示 |
| 报名表下拉选项 | 投递组别按比赛配置；基础风格来自比赛绑定风格库 |
| 评分表 | 专业评审固定 5 项；跨界评审按比赛配置；默认总分 50 |
| 评分输入 | 分数必须为整数，每个维度下方有备注框，字数按所有备注合计校验 |
| 评审匿名信息 | 展示匿名编号、组别、风格、ABV 和允许公开的额外字段 |
| 隐藏信息 | 不展示酒名、厂商、联系人和其他可识别厂商的信息 |
| 桌长个人评分 | 桌长先按专业评审表提交个人评分 |
| 普通评审修改 | 桌长汇总或本桌确认前可修改 |
| 桌长汇总 | 查看原始评分，填写共识分、综合评语，勾选晋级 |
| 同桌确认 | 桌长最终提交前，同桌评审确认共识分、评语和晋级名单 |
| 晋级 | 发生在轮次/评审桌级别，数量按每桌配置 |
| 后续轮 | 使用排序模式，可支持 `TOP_N`、`MEDALS`、`CHAMPION` |
| 轮次锁定 | 主办方锁定后作为后续轮或奖项来源 |
| 现场看板 | 已有入口，需验收大屏指标和展示口径 |
| 分数排名 | 避免跨桌直接排名，正式奖项通过排序轮和主办方确认 |
| 厂商结果可见 | 发布后展示评分、备注、桌长总结、轮次结果、奖项和证书 |
